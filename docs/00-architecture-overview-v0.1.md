## 0. 이 문서의 위치

이 문서는 **운명 플랫폼 전체의 상위 설계**를 규정한다. 하위 문서(엔진 스펙, 디자인, 백엔드 등)는 모두 이 문서의 계약을 따른다. 따라서 이 문서에서 결정된 사항을 개별 문서가 임의로 뒤집을 수 없다. 변경이 필요하면 이 문서를 먼저 개정한 후 하위 문서를 맞춘다.

**서술 원칙.**
- 구현 가능한 구체성(concrete). 추상적 미덕 선언 금지.
- 결정(Decision)은 반드시 **근거(Why)** 와 **영향 범위(Scope)** 명시.
- 미결정 사항은 `§ Open Questions`에 남기고, 결정 데드라인 기재.

## 1. 제품 개요

### 1.1 한 줄 정의

**"5개의 서로 다른 체계가 한 사람을 어떻게 설명하는지 한 화면에서 겹쳐 보여주는 서비스."**

### 1.2 핵심 사용자 행위 (Top Jobs)

| # | 사용자가 하고 싶은 일 | 완료 신호 |
|---|---|---|
| J1 | "나를 총체적으로 이해하고 싶다" — 사주+MBTI+애니어그램+별자리가 공통으로 말하는 키워드가 뭔지 보고 싶다 | 종합 리포트 1회 열람 |
| J2 | "연인/상사/가족을 이해하고 싶다" — 상대 프로필로 교차 분석 | 궁합 리포트 1회 열람 |
| J3 | "오늘/이번 달이 어떤가" — 시간성 조회 | 데일리/먼슬리 카드 확인 |
| J4 | "내 결과를 남에게 보여주고 싶다" — 프로필 카드 공유 | 공유 링크 생성/SNS 업로드 |
| J5 | "구체 고민에 조언받고 싶다" — 질문형 상담 | 리포트 기반 Q&A |

J1, J2, J3가 **핵심 3축**. J4는 바이럴 루프, J5는 업셀·리텐션 축.

### 1.3 타깃 사용자

- **1차:** 20~40대 남녀. 명리·MBTI·애니어그램 중 최소 하나 이상은 들어본 적 있는 일반 대중.
- **톤 매칭:** 점신처럼 "도사 톤"도 아니고, 헬로우봇처럼 "친구 톤"도 아닌, **에디토리얼한 지식 서비스 톤** (노션 템플릿, The Pudding, Arc 느낌).

### 1.4 NOT goals (v1 범위 밖)

- 실제 감정원·상담가 매칭(인력 마켓플레이스) — v3+
- 부적·개운·택일 등 주술 상품 — 영구 제외(브랜드 톤과 충돌)
- 연애 매칭(소개팅) — 영구 제외
- 주식·경마·로또 등 사행적 예측 — 영구 제외(정책 리스크)

> 이 4개는 명시적으로 **영구 제외** 또는 **장기 후순위**. 기획/마케팅 의사결정 시 "이거 부적 상품 같지 않나?" 질문을 거쳐 자체 검열.

## 2. 포지셔닝과 차별화

### 2.1 시장 포지션 매트릭스

```
                    전통/전문성 강함
                          ▲
                          │
   [원광만세력]      ●    │    ●    [점신]
   [천을귀인]             │
                          │
   ◀ 도구 중심 ───────────┼─────────── 해석·엔터테인먼트 중심 ▶
                          │
                          │    ●  [헬로우봇]
   [MBTI 16p]       ●     │    ●  [포스텔러]
                          │    ●  [Co-Star]
                          ▼
                    캐주얼/엔터테인먼트 강함
```

**우리의 자리:** **가로축 중앙 + 세로축 상단** — "도구로서의 정확성"과 "해석으로서의 풍부함"을 **지적이고 미니멀한 디자인**으로 통합.

### 2.2 핵심 차별화 3요소

**D1. 교차 해석.** 단일 체계가 아니라 5개 엔진이 동시에 한 사람을 설명하고, 그 교집합/차이를 노출한다. 이것이 브랜드의 레종데트르.

**D2. 투명한 계산.** 만세력 스펙이 이미 규정한 `trace` 개념을 전 엔진에 확장. 모든 결과에 "왜 이 결과인가" 펼쳐보기 가능. 점신·포스텔러 모두 이걸 안 준다.

**D3. 에디토리얼 디자인.** 운세 앱에서 흔한 파스텔·캐릭터·이모지 스팸 대신, 세리프 타이포그라피 + 절제된 그리드 + 데이터 시각화. 20~40대 지적 소비자 타깃.

### 2.3 벤치마크 요약

| 서비스 | 강점 | 우리가 가져올 것 | 우리가 버릴 것 |
|---|---|---|---|
| 점신 | 전문성, 만세력 정확도 | 계산 정확도, 용어 정확성 | 도사 톤, 광고 밀도 |
| 포스텔러 | 친절한 설명, 차트 UI | 부드러운 설명, 시각화 | 별자리만 다룸, 한정적 깊이 |
| Co-Star | 에디토리얼 톤, 타이포 | 비주얼 언어, 짧고 시적인 카피 | 부정적 멘트, 오컬트 느낌 |
| The Pattern | 교차 관계 분석 | 관계 궁합 UX | 복잡한 용어, 긴 설명 |
| Notion | 정보 아키텍처 | 블록 기반 리포트 구성 | — |

## 3. 시스템 레이어

```
┌──────────────────────────────────────────────────────────────────┐
│ L7  Presentation       웹 UI · PWA · (차후) 네이티브 앱           │
├──────────────────────────────────────────────────────────────────┤
│ L6  Experience         온보딩, 입력 폼, 리포트 뷰어, 결제 UX       │
├──────────────────────────────────────────────────────────────────┤
│ L5  Interpretation     교차 해석기: 엔진 출력 → 자연어 리포트      │
├──────────────────────────────────────────────────────────────────┤
│ L4  Orchestration      엔진 파이프라인, 캐시, 버전 관리            │
├──────────────────────────────────────────────────────────────────┤
│ L3  Engines            명리 · 자미두수 · Goldschneider · MBTI · Enneagram │
├──────────────────────────────────────────────────────────────────┤
│ L2  Domain Primitives  Stem/Branch, 별자리, 유형 enum, 시간, 좌표 │
├──────────────────────────────────────────────────────────────────┤
│ L1  Platform Services  인증·DB·스토리지·결제·관측(Observability)  │
├──────────────────────────────────────────────────────────────────┤
│ L0  Infra              클라우드, CDN, DNS, 모니터링                │
└──────────────────────────────────────────────────────────────────┘
```

### 3.1 레이어별 책임

| 레이어 | 책임 | 금지 |
|---|---|---|
| L0 Infra | 가용성, 오토스케일, 로그 수집 | 비즈니스 로직 |
| L1 Platform | 인증, 세션, PG연동, DB 스키마, 로깅 | 엔진 계산, 해석 텍스트 |
| L2 Primitives | 타입·상수·순수함수 | 상태, I/O |
| L3 Engines | 순수 계산: 입력→출력+trace | 자연어 생성, 저장 |
| L4 Orchestration | 여러 엔진 호출, 병렬화, 캐싱, 스키마 버전 관리 | 해석 문장 생성 |
| L5 Interpretation | 엔진 출력을 교차 조합→리포트 DTO 생성 | DOM 조작, 스타일 |
| L6 Experience | 사용자 입력 수집·검증, 상태 관리, 결제 플로우 | 계산 |
| L7 Presentation | 픽셀 단위 렌더링, 애니메이션, 접근성 | 비즈니스 판정 |

### 3.2 불변 규칙

- **L3 엔진은 항상 순수함수.** 같은 입력 → 같은 출력. I/O 금지. (만세력 스펙 §0 원칙 계승)
- **L3는 L5에 의존하지 않는다.** 엔진이 "해석 텍스트"를 반환하지 않는다. 엔진은 숫자·enum·구조체만.
- **L5는 L3의 `trace`까지 읽을 수 있다.** 이유: "왜 이런 해석이 나왔나"를 역으로 설명해야 하므로.
- **크로스 레이어 호출 금지.** L7이 L3를 직접 호출할 수 없다. 반드시 L6 → L4 → L3.

## 4. 엔진 카탈로그

### 4.1 엔진 요약표

| ID | 엔진 | 입력(핵심) | 출력(핵심) | 계산 난이도 | 문서 |
|---|---|---|---|---|---|
| E1 | 명리 (만세력) | 생년월일시 + 출생지 | 4주 8자 + 지장간 + trace | **매우 높음**(천문) | `10-engine-manseryeok` |
| E2 | 자미두수 | 음력 생년월일시 + 성별 | 12궁 + 14주성 + 보조성 배치 | **높음** | `11-engine-ziweidoushu` |
| E3 | Goldschneider 48주 | 양력 생년월일 | 48 week-archetype + cusp 정보 | **낮음**(테이블 매핑) | `12-engine-goldschneider` |
| E4 | MBTI | 설문 응답 (자체 문항 60~90개) | 16유형 + 4축 강도 | **낮음** | `13-engine-mbti` |
| E5 | Enneagram | 설문 응답 (자체 문항 135개) | 9유형 + 윙 + 통합/분열 방향 | **낮음** | `14-engine-enneagram` |

### 4.2 엔진별 입력 데이터 교집합

```
                    생년월일     시각       출생지     설문응답   성별
E1 명리              ✅          ✅         ✅         ❌         △*
E2 자미두수          ✅(음력)    ✅         ❌         ❌         ✅
E3 Goldschneider     ✅(양력)    ❌         ❌         ❌         ❌
E4 MBTI              ❌          ❌         ❌         ✅         ❌
E5 Enneagram         ❌          ❌         ❌         ✅         ❌

* 명리는 남녀 대운 방향이 다름
```

**시사점.** E1+E2+E3는 **생년월일시/출생지 단일 입력 폼**으로 모두 돌릴 수 있다. E4·E5는 별도 **설문 플로우**. 온보딩은 이 두 경로로 분리한다.

### 4.3 엔진 인터페이스 일반형

모든 엔진은 아래 형태의 순수함수로 export한다.

```typescript
interface Engine<Input, Output> {
  readonly id: EngineId;            // 'manseryeok' | 'ziweidoushu' | ...
  readonly schemaVersion: string;   // '1.0.0' — 출력 스키마 버전
  compute(input: Input): Output;    // 동기. 외부 호출 없이 완결.
}
```

- 비동기는 오케스트레이션 층(L4)이 책임. 엔진 자체는 동기·순수.
- 모든 출력에 `trace` 필드 포함. 크기 우려 시 `trace: { summary, detail? }` 2단계 구조 허용.
- 스키마 버전은 엔진 출력 DTO의 브레이킹 변경 시에만 메이저 올림. 해석 로직 변경은 영향 없음.

### 4.4 엔진별 정확도·완성도 목표

| 엔진 | v1 목표 | v2+ 목표 |
|---|---|---|
| E1 명리 | 점신 대비 99.5%+ (기존 스펙 §1.3) | — |
| E2 자미두수 | 주요 앱(紫微斗數排盤) 3종 대비 99%+ | 추가 보조성·사화 |
| E3 Goldschneider | 원서 대비 100% 일치 (테이블이므로) | 도서 1365개 일자별 해설 매핑 |
| E4 MBTI | 공식 MBTI와의 실시 시험 상관 0.75+ | 자체 문항 재검수 |
| E5 Enneagram | RHETI 135문항 기반 구현 | 자체 간이 27문항 병행 |

> **주의.** MBTI와 Enneagram은 **공식 기관의 상표권 이슈**가 있다. v1에서는 "MBTI 유형 측정"이 아니라 **"16개 성격 유형 측정"** 같은 회피 표현을 사용. 법무 검토 이전까지 공식 로고·약어 표기 금지. 상세는 `60-privacy-compliance`에서 다룸.

## 5. 데이터 흐름

### 5.1 사용자 1인 전체 플로우

```
[회원가입/로그인]
       │
       ▼
[프로필 생성] ──────── (입력) 생년월일, 시, 출생지, 성별, 양/음력, 자시 관례
       │
       ▼
[엔진 E1·E2·E3 실행] ─ 병렬, L4 오케스트레이션
       │
       ▼
[중간 결과 캐시] ─── DB + CDN 엣지 캐시
       │
       ▼
[설문 진입 동기 부여 UX] ─── "MBTI·애니어그램 추가하면 교차 해석 가능"
       │
       ▼
[설문 E4·E5 실행] ─── 설문 세션은 중단·재개 가능
       │
       ▼
[L5 교차 해석기] ──── 5개 엔진 결과 읽어 리포트 DTO 생성
       │
       ▼
[리포트 뷰어(L7)] ─── 섹션별 점진 로드
       │
       ▼
[업셀 지점] ──────── 상세 리포트(PDF), 궁합, 월별 상세
```

### 5.2 데이터 주체별 수명

| 데이터 | 생성 | 수명 | 삭제 주체 |
|---|---|---|---|
| 생년월일시/출생지 (PII) | 프로필 생성 시 | 탈퇴 시까지 | 사용자 요청 시 즉시, 탈퇴 30일 보관 후 삭제 |
| 엔진 계산 결과 (파생) | 엔진 실행 시 | 엔진 스키마 버전 유지 기간 | 스키마 버전 업 시 재계산 |
| 리포트 텍스트 (파생) | 해석기 실행 시 | 사용자 열람 시까지 | 사용자 요청 시 |
| 결제 기록 | 결제 시 | 법정 보존기간 (전자상거래법 5년) | 법정 기한 도래 시 |
| 설문 응답 원본 | 설문 제출 시 | 탈퇴 시까지 (재측정 옵션 제공) | 탈퇴 시 |

### 5.3 캐시 키 설계

```
engine_result_cache_key = hash(engineId + schemaVersion + canonicalInput)
```

- `canonicalInput`: 입력 객체의 JSON canonical form (키 정렬 + 필드 정규화).
- 캐시 히트 시 재계산 생략. 스키마 버전이 바뀌면 자동 무효화.
- Redis TTL: 사용자 소유 결과 무제한(유료), 익명 미리보기 24h.

### 6.1 결정 요약표

### 6.2 결정 논리 — Flutter/React Native 비선택 사유

이전 턴의 옵션 비교와 이어짐. 결정: **웹 선행 → PWA → 차후 앱 래핑(Capacitor 또는 RN WebView)**.

- **웹 선행 확정.** 사용자가 명시.
- **앱은 나중.** 5월 MVP에 RN/Flutter 동시 유지는 1인으로 과부하.
- **Capacitor** 또는 **React Native WebView**로 v1.5~v2에서 앱화. 그때까지는 PWA(홈 화면 추가, 오프라인 최소) 수준.
- 장기적으로 **네이티브 앱 필요 시점:** 푸시 알림이 핵심 리텐션 장치가 될 때, 앱스토어 인앱결제 수수료가 구독 비즈니스에 결정적일 때.

### 6.3 모델/AI 의존성

해석 텍스트 생성(L5 Interpretation)은 **하이브리드** 전략:
- **템플릿 기반 생성**: 엔진 출력 → 사전 작성 조각 합성. 빠름·결정론적·무료.
- **LLM 보조**: 템플릿으로 안 되는 자연스러운 톤 조정, Q&A 상담(J5). Anthropic Claude API (`claude-sonnet-4-6`) 1차, 필요 시 `claude-opus-4-7`.
- **프롬프트 캐싱 필수**. 엔진 출력이 길어 컨텍스트 재사용 큼.
- MVP는 **템플릿 100%**. LLM은 v1.5 이후.

이유: LLM 의존 시 (1) 응답 지연, (2) 비용 변동, (3) 결정론성 상실. 초기 품질 관리 어렵다.

## 10. 보안·프라이버시 기본 원칙

상세는 `60-privacy-compliance-spec`.

**원칙 5개:**

1. **개인정보 최소 수집.** 생년월일시/출생지는 서비스 본질이라 불가피. 그 외는 수집하지 않는다. 이메일도 소셜 로그인으로 대체 권장.
2. **가능한 한 클라이언트 계산.** 만세력 스펙 §12.2 계승. PII를 서버로 보내지 않는 경로를 기본값으로.
3. **저장 시 분리.** PII(생년월일시)와 결과(엔진 출력)을 다른 테이블/스키마에 저장, 결과는 해시 키로 조회.
4. **로그에 PII 금지.** 에러 리포트·로그에는 해시된 입력만.
5. **탈퇴는 즉시 반영.** 탈퇴 버튼 누르면 소프트 삭제, 30일 후 하드 삭제. 결제 기록은 법정 보존.

## 11. 품질 기준과 검증

### 11.1 엔진 품질

만세력 스펙 §11의 검증 전략을 **전 엔진에 일반화**:

- **Golden Dataset**: 각 엔진당 100건+. 기존 상용 서비스와 교차검증.
- **Property-based test**: 불변 속성(예: MBTI 동일 응답 → 동일 결과)을 fast-check로.
- **경계 케이스 카탈로그**: 엔진별로 `tests/boundary-cases.json`.

### 11.2 UI 품질

- **Playwright e2e**: 핵심 플로우 5개 (회원가입 → 프로필 → 리포트 → 결제 → 공유).
- **Visual regression**: Chromatic 또는 Percy는 비용상 v2. v1은 스크린샷 수동.

### 11.3 해석 품질

- 템플릿 조각마다 **도메인 검증자 리뷰**. v1은 주당 5시간 외주 전문가 1명.
- 사용자 피드백 수집: 리포트 섹션마다 "이 해석 맞았나요?" 토글. 데이터로 누적.
